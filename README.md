# Архитектура системы

Проект реализован в виде микросервиса обработки событий (Event Processor), который принимает данные из внешних систем, объединяет их и обеспечивает гарантированную доставку в систему AdVantage.

## Общая схема взаимодействия

```text
ADs Service
      |
      v
 POST /click
      |
      v
+------------------+
| Event Processor  |
+------------------+
      ^
      |
 POST /payment
      |
      |
Payment Service

      |
      v
 PostgreSQL
      |
      v
 Matching Service
      |
      v
 Outbox
      |
      v
 Retry Worker
      |
      v
 AdVantage
```

## Компоненты системы

### API Layer

Слой REST API, реализованный на FastAPI.

Предоставляет следующие endpoints:

* POST /click — получение информации о клике от ADs Service;
* POST /payment — получение информации о покупке от Payment Service;
* GET /health — проверка состояния сервиса.

Задача слоя — принять запрос, выполнить базовую валидацию и передать данные в сервисный слой.

---

### Service Layer

Содержит бизнес-логику приложения.

Основные сервисы:

#### ClickService

Отвечает за обработку кликов.

Функции:

* проверка существования клика по clid;
* защита от дублирующихся кликов;
* сохранение нового клика.

#### PaymentService

Отвечает за обработку покупок.

Функции:

* проверка уникальности покупки по паре (clid, ts);
* сохранение новой покупки.

#### MatchingService

Выполняет связывание кликов и покупок.

Функции:

* поиск клика по clid;
* поиск покупки по clid;
* формирование итогового события для AdVantage;
* создание записи в Outbox.

---

### Repository Layer

Слой доступа к данным.

Реализует взаимодействие с PostgreSQL через SQLAlchemy.

Репозитории:

* ClickRepository;
* PaymentRepository;
* OutboxRepository.

Использование репозиториев позволяет отделить бизнес-логику от работы с базой данных.

---

### PostgreSQL

Используется для постоянного хранения данных.

Основные таблицы:

#### clicks

Хранит информацию о кликах.

Поля:

* id;
* clid;
* ad_id;
* click_spend;
* ts.

#### payments

Хранит информацию о покупках.

Поля:

* id;
* clid;
* payout;
* ts;
* processed.

#### outbox

Хранит события, ожидающие отправки в AdVantage.

Поля:

* id;
* payload;
* status;
* retry_count;
* next_retry_at.

---

### Outbox Pattern

Для обеспечения гарантированной доставки используется паттерн Outbox.

После успешного сопоставления Click и Payment событие сначала сохраняется в таблицу outbox.

Только после этого оно может быть отправлено во внешнюю систему.

Такой подход предотвращает потерю данных при сбоях сети, недоступности внешнего сервиса или перезапуске приложения.

---

### Retry Worker

Фоновый процесс, запускаемый через APScheduler.

Функции:

* поиск необработанных событий;
* отправка событий в AdVantage;
* повторные попытки доставки при ошибке;
* увеличение retry_count;
* планирование следующей попытки отправки.

---

### AdVantage Client

Отдельный компонент, отвечающий за взаимодействие с внешней системой AdVantage.

Функции:

* формирование HTTP-запроса;
* отправка данных;
* обработка ответа;
* возврат результата отправки.

---

## Используемые паттерны проектирования

### Repository Pattern

Инкапсулирует доступ к данным и изолирует бизнес-логику от SQLAlchemy.

### Service Layer Pattern

Выделяет бизнес-логику в отдельный слой сервисов.

### DTO Pattern

Используется для передачи данных между API и бизнес-логикой.

### Outbox Pattern

Гарантирует отсутствие потери событий при сбоях внешних систем.

### Retry Pattern

Обеспечивает повторные попытки доставки при ошибках.

### Eventual Consistency

Система корректно обрабатывает ситуацию, когда покупка приходит раньше клика. После появления недостающих данных событие автоматически формируется и передаётся в Outbox.

## Обеспечение надёжности

Для выполнения ключевого требования задания — гарантированной доставки данных в AdVantage — используются следующие механизмы:

* хранение событий в PostgreSQL;
* паттерн Outbox;
* Retry Worker;
* повторные попытки отправки;
* защита от дублирующихся кликов;
* уникализация покупок по паре (clid, ts);
* поддержка произвольного порядка поступления событий.

В результате система обеспечивает надёжную обработку и доставку данных даже при временной недоступности внешнего сервиса AdVantage.

## Проверка работоспособности проекта

### 1. Запуск PostgreSQL

Запустить контейнер с базой данных:

```bash
docker compose up -d
```

Проверить, что контейнер успешно запущен:

```bash
docker ps
```

---

### 2. Запуск приложения

Запустить FastAPI-приложение:

```bash
uv run uvicorn app.main:app --reload --port 8001
```

После успешного запуска будут доступны:

Swagger UI:

```
http://localhost:8001/docs
```

Проверка состояния сервиса:

```
http://localhost:8001/health
```

Ожидаемый ответ:

```json
{
  "status": "ok"
}
```

---

### 3. Проверка сохранения клика

В Swagger выполнить запрос:

POST /click

Тело запроса:

```json
{
  "clid": "test_clid_1",
  "ad_id": 123,
  "click_spend": 2.5,
  "ts": "2026-06-13T10:00:00"
}
```

Проверить наличие записи в базе данных:

```sql
SELECT * FROM clicks;
```

Ожидаемый результат:

В таблице должна появиться одна запись с clid = 'test_clid_1'.

---

### 4. Проверка сохранения покупки

В Swagger выполнить запрос:

POST /payment

Тело запроса:

```json
{
  "clid": "test_clid_1",
  "payout": 100,
  "ts": "2026-06-13T11:00:00"
}
```

Проверить наличие записи:

```sql
SELECT * FROM payments;
```

Ожидаемый результат:

В таблице должна появиться запись с clid = 'test_clid_1'.

---

### 5. Проверка механизма сопоставления событий

После получения Click и Payment с одинаковым clid сервис автоматически выполняет сопоставление событий.

Проверка:

```sql
SELECT
    c.clid,
    c.click_spend,
    p.payout,
    p.processed
FROM clicks c
JOIN payments p
    ON c.clid = p.clid;
```

Ожидаемый результат:

```text
test_clid_1 | 2.5 | 100 | true
```

Значение processed = true означает, что покупка была успешно обработана.

---

### 6. Проверка создания события для AdVantage

После успешного сопоставления событий должна появиться запись в таблице Outbox.

Проверка:

```sql
SELECT * FROM outbox;
```

Ожидаемый результат:

В таблице должна присутствовать запись со статусом PENDING.

---

### 7. Проверка гарантированной доставки

Сервис AdVantage в рамках задания недоступен, поэтому отправка завершается ошибкой.

Проверка:

```sql
SELECT
    id,
    status,
    retry_count,
    next_retry_at
FROM outbox;
```

Ожидаемый результат:

```text
id | status  | retry_count
--------------------------
1  | PENDING | 1
```

Через некоторое время значение retry_count должно увеличиваться.

Это подтверждает работу механизма повторных попыток отправки.

---

### 8. Проверка идемпотентности

Повторно отправить тот же запрос Click:

```json
{
  "clid": "test_clid_1",
  "ad_id": 123,
  "click_spend": 2.5,
  "ts": "2026-06-13T10:00:00"
}
```

Проверить количество записей:

```sql
SELECT *
FROM clicks
WHERE clid = 'test_clid_1';
```

Ожидаемый результат:

В таблице должна остаться только одна запись.

---

### 9. Проверка Eventual Consistency

Очистить таблицы:

```sql
TRUNCATE clicks, payments, outbox RESTART IDENTITY;
```

Сначала отправить Payment:

```json
{
  "clid": "eventual_test",
  "payout": 200,
  "ts": "2026-06-13T12:00:00"
}
```

Затем отправить Click:

```json
{
  "clid": "eventual_test",
  "ad_id": 456,
  "click_spend": 5,
  "ts": "2026-06-13T11:00:00"
}
```

Проверить наличие записи в Outbox:

```sql
SELECT * FROM outbox;
```

Ожидаемый результат:

Событие должно быть успешно создано несмотря на то, что Payment был получен раньше Click.

---

### Итог

* приложение успешно запускается;
* данные сохраняются в PostgreSQL;
* Click и Payment сопоставляются по clid;
* создаются записи в Outbox;
* работает механизм повторных попыток отправки;
* повторные Click-запросы не создают дубликатов;
* порядок поступления событий не влияет на результат обработки.
